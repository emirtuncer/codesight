# CLAUDE.md

## What This Is

codesight is a Go CLI that creates Obsidian-compatible Markdown files from codebases using tree-sitter. It parses 7 languages, generates package-level abstractions with API surfaces, dependency graphs, and PRD-style feature specs. The `.codesight/` folder serves as a knowledge layer for AI coding agents.

## Building & Testing

CGO is required (tree-sitter is a C library). On Windows, use MSYS2 MinGW gcc.

```bash
# Build
CGO_ENABLED=1 go build -o codesight.exe ./cmd/codesight/

# Windows with MSYS2
CC=C:/msys64/mingw64/bin/gcc.exe CGO_ENABLED=1 go build -o codesight.exe ./cmd/codesight/

# Run all tests
CGO_ENABLED=1 go test ./... -count=1

# Run a single test
CGO_ENABLED=1 go test -run TestName ./internal/sync/ -count=1
```

The `-count=1` flag disables test caching.

## Architecture

### Pipeline: Discover → Scan → Parse → Generate

On `init` and `sync`, manifest detection (`internal/discover/`) first walks the directory tree looking for project manifests (`go.mod`, `package.json`, `Cargo.toml`, etc.). Each outermost manifest defines a sub-project boundary. The sync engine then runs once per discovered project:

1. **Scan** (`internal/scanner/`) — walks directory tree respecting nested `.gitignore` files, SHA256 hashes each file, detects language by extension. Skips testdata, vendor, node_modules, etc.
2. **Parse** (`internal/parser/`) — tree-sitter parses files into ASTs. Embedded S-expression queries (`.scm` files in `queries/`) extract symbols. Language-specific walkers (`internal/parser/walkers/`) handle language-specific patterns (generics, decorators, trait impls).
3. **Generate** (`internal/sync/`) — groups symbols by package (last directory segment), writes one MD per package with API surface, types, dependencies, and Claude-managed sections.

After the pipeline: feature detection matches file paths against configurable regex patterns, changelog tracks symbol-level changes, architecture stubs are created.

### Output: `.codesight/` Vault

```
.codesight/
  <project>/
    packages/          # one MD per package (the core abstraction)
    features/          # PRD-style feature specs with requirements
    architecture/      # system design docs (Claude-managed)
    tasks/             # project tasks with IDs, urgency, status
    _changelog.md      # symbol-level change history
    _index.md
  _config.md           # settings, feature/ignore patterns, ID counters
```

### Package MD Format

Each package MD has two zones separated by `---`:
- **Tree-sitter sections** (above) — API Surface, Types, Constants, Dependencies, Files. Regenerated by `sync`.
- **Claude-managed sections** (below) — Overview, Architecture Notes, Usage Examples, Gotchas, Tasks. Preserved across syncs, filled by `analyze`.

### Key Packages

| Package | Purpose |
|---------|---------|
| `internal/discover/` | Manifest detection for sub-project boundaries |
| `internal/parser/` | Tree-sitter engine, query runner, language walkers |
| `internal/scanner/` | File walking + gitignore + hashing |
| `internal/markdown/` | Parse/write/update MD files with strict schemas |
| `internal/search/` | Load all MDs, query filter, dependency graph |
| `internal/sync/` | Orchestrate scan→parse→generate, changelog, features |
| `internal/tasks/` | Task CRUD with auto-increment IDs |
| `internal/mcp/` | MCP server (5 tools: search, status, task create/update, sync) |
| `internal/cli/` | Cobra commands: init, sync, search, task, status, analyze, serve, feature |
| `queries/` | Tree-sitter S-expression queries (.scm) embedded via go:embed |

### Key Design Decisions

- **Package-level, not symbol-level** — one MD per package instead of per-symbol. API surfaces list exported functions, internal functions are just names.
- **Two-zone MDs** — tree-sitter owns the top (regenerated), Claude owns the bottom (preserved). `SplitAtClaudeDivider()` and `UpdatePackage()` handle merging.
- **Incremental everything** — sync compares content hashes to skip unchanged packages. Analyze tracks `analysis_hash` to skip already-analyzed items.
- **Config in Markdown** — `_config.md` uses YAML frontmatter for settings (feature patterns, ignore patterns, ID counters). No JSON/TOML files.
- **Feature patterns** — user-defined regex in config, capture group 1 = feature name. Applied alongside built-in heuristics (modules/, features/, handlers/).
- **Graceful degradation** — parse errors skip files, missing claude CLI returns helpful message, vault works as plain files even without the CLI.
- **Error wrapping** — `fmt.Errorf("operation: %w", err)` convention throughout.

### Claude Code Integration

`init` generates:
- `.claude/settings.json` — hooks (SessionStart: `codesight status`, PostToolUse after git commit: `codesight sync`)
- `.claude/skills/codesight.md` — skill file teaching search, task, and status commands

Claude can also read `.codesight/` MDs directly as a fallback.
